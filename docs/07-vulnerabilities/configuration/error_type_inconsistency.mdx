# Error type inconsistency

## Description

An error type inconsistency occurred.


## Remediation

An error type inconsistency can be caused by a misconfiguration of the GraphQL server.
To fix this issue, return a contextual error type for each error instead of an Internal Server Error.

For example, if the request fails during the validation phase, return a validation error.
By default, it will return an Internal Server Error.



### GraphQL Specific

<details>
    <summary>Apollo</summary>

Ensure that error types are consistent across your Apollo server implementation. Define a custom error format if necessary.

</details>

<details>
    <summary>Yoga</summary>

In Yoga, standardize your error handling by creating a consistent error format and using it throughout your resolvers.

</details>

<details>
    <summary>Awsappsync</summary>

For AWS AppSync, use the $util.error function in your VTL templates to throw errors with a consistent structure.

</details>

<details>
    <summary>Graphqlgo</summary>

In GraphQL Go, define a custom error type that implements the `error` interface and use it consistently across your resolvers.

</details>

<details>
    <summary>Graphqlruby</summary>

In GraphQL Ruby, use the `GraphQL::ExecutionError` class to create errors with a consistent type and message format.

</details>

<details>
    <summary>Hasura</summary>

For Hasura, ensure that custom errors returned from actions or remote schemas have a consistent format.

</details>
